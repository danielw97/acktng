# Proposal: LLM-Powered NPC Dialogue

## Overview

Allow players to converse with NPCs via the `say` command. When a player speaks
in a room containing a dialogue-enabled NPC, the NPC's response is generated by
a local LLM via OpenClaw (an OpenAI-compatible local model server). Responses
are delivered asynchronously so the game loop is never blocked.

---

## Architecture

### High-Level Flow

```
Player types: say hello blacksmith
      |
      v
do_say() in act_comm.c
      |
      +-- loop over room NPCs
      |      |
      |      +-- IS_NPC(ppl) && IS_SET(ppl->act, ACT_AI_DIALOGUE)?
      |               |
      |               v
      |         npc_dialogue_dispatch(npc, player, message)
      |           - look up or create conversation history
      |           - check expiry; clear if stale
      |           - serialize history snapshot into request struct
      |           - append user turn to live history
      |           - push request to REQUEST QUEUE
      |           - return immediately (game loop unblocked)
      |
WORKER THREAD
      |
      +-- dequeue request
      +-- POST to OpenClaw (blocking HTTP via libcurl)
      +-- receive response text
      +-- push to RESPONSE QUEUE
      |
GAME LOOP (next tick)
      |
      v
update_handler() in update.c
      |
      +-- npc_dialogue_deliver()
            - drain RESPONSE QUEUE
            - validate NPC and player still exist and share a room
            - append assistant turn to live history
            - call do_say(npc, response_text)
```

## Components

### 1. NPC Dialogue Flag

Add `ACT_AI_DIALOGUE` to the mob act flags (in `typedefs.h`). Only NPCs with
this flag set will respond to player speech. The flag is set in the area file
using the standard mob `Act:` field.

This prevents every mob in a room from attempting to respond and limits LLM
calls to intentionally dialogue-enabled characters.

### 2. NPC Persona / System Prompt

Each dialogue-enabled NPC needs a system prompt describing its persona. This is
passed to the LLM as the `system` role message and should cover:

- Who the NPC is (name, role, location)
- Personality and speech style
- What the NPC knows and does not know
- Any quest-relevant information they can share

**Storage:** A new field `char *ai_prompt` on `CHAR_DATA`, populated from the
area file. The area file format gains a new optional section for AI-enabled mobs:

```
#MOBILES
...
Name       Goran the blacksmith~
...
AiPrompt
You are Goran, the blacksmith of Midgaard. You are gruff and practical, speaking
in short sentences. You know about weapons, armor, and local rumors. You do not
know about events outside the city. Never break character.
~
End
```

The `ai_prompt` string is copied into the request at dispatch time so the worker
thread never holds a pointer into live game memory.

### 3. Threading Model

The game loop is single-threaded. All blocking I/O happens on a dedicated worker
thread. Communication between threads uses two mutex-protected queues.

```
GAME LOOP THREAD              WORKER THREAD
─────────────────             ─────────────
dispatch → REQUEST QUEUE ──→  dequeue → HTTP call (blocks here)
                              push → RESPONSE QUEUE
tick: drain RESPONSE QUEUE ←─ (written by worker)
  → do_say(npc, text)
```

**Rules:**
- The worker thread **never touches game structs** (`CHAR_DATA`, `ROOM_INDEX_DATA`, etc.)
- The game loop **never blocks** on network I/O
- All data passed to the worker is **copied by value** into the request struct
- `curl_global_init()` is called once at startup before the worker thread is spawned

### 4. Request and Response Structs

```c
#define MAX_DIALOGUE_TURNS   8
#define MAX_REQUEST_TURNS    9   /* history + the new user message */

typedef struct dialogue_turn_copy {
    char role[16];       /* "user" or "assistant" */
    char content[512];
} DIALOGUE_TURN_COPY;

typedef struct npc_dialogue_request {
    CHAR_DATA         *npc;
    CHAR_DATA         *player;
    char               player_name[50];
    char               system_prompt[1024];
    int                history_count;
    DIALOGUE_TURN_COPY history[MAX_REQUEST_TURNS];  /* serialized snapshot */
    struct npc_dialogue_request *next;
} NPC_DLG_REQ;

typedef struct npc_dialogue_response {
    CHAR_DATA *npc;
    CHAR_DATA *player;
    char       response_text[512];
    struct npc_dialogue_response *next;
} NPC_DLG_RESP;
```

### 5. Queues and Synchronization

```c
/* Request queue: game loop writes, worker reads */
pthread_mutex_t  req_mutex  = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t   req_cond   = PTHREAD_COND_INITIALIZER;
NPC_DLG_REQ     *req_head   = NULL;

/* Response queue: worker writes, game loop reads */
pthread_mutex_t  resp_mutex = PTHREAD_MUTEX_INITIALIZER;
NPC_DLG_RESP    *resp_head  = NULL;
```

The worker blocks on `pthread_cond_wait` when the request queue is empty. Queue
node memory uses `malloc`/`free` directly — **not** the MUD's custom allocator
(`ssm.c`), which is not thread-safe.

### 6. Worker Thread

```c
void *npc_dialogue_worker(void *unused)
{
    while (1)
    {
        pthread_mutex_lock(&req_mutex);
        while (req_head == NULL)
            pthread_cond_wait(&req_cond, &req_mutex);
        NPC_DLG_REQ *req = req_head;
        req_head = req->next;
        pthread_mutex_unlock(&req_mutex);

        char response_text[512];
        call_openclaw(req, response_text, sizeof(response_text));

        NPC_DLG_RESP *resp = malloc(sizeof(NPC_DLG_RESP));
        resp->npc    = req->npc;
        resp->player = req->player;
        strncpy(resp->response_text, response_text, sizeof(resp->response_text) - 1);
        resp->response_text[sizeof(resp->response_text) - 1] = '\0';

        pthread_mutex_lock(&resp_mutex);
        resp->next = resp_head;
        resp_head  = resp;
        pthread_mutex_unlock(&resp_mutex);

        free(req);
    }
    return NULL;
}
```

### 7. OpenClaw HTTP Call

The worker calls `call_openclaw()`, which uses libcurl to POST to the local
OpenClaw endpoint. The request body is an OpenAI-compatible chat completions
payload:

```json
{
  "model": "your-local-model",
  "messages": [
    { "role": "system",    "content": "<npc persona>" },
    { "role": "user",      "content": "what are you selling?" },
    { "role": "assistant", "content": "Weapons and armor, traveler." },
    { "role": "user",      "content": "how much for the sword?" }
  ],
  "max_tokens": 100,
  "temperature": 0.7
}
```

The player's name is prepended to user messages (`"Arathorn: how much for the
sword?"`) so the NPC can address them by name.

A short timeout (e.g. 5 seconds) is set on the curl handle. On timeout or HTTP
error, the response is discarded silently — the player simply gets no reply.

### 8. Game Loop Integration

**Dispatch — `do_say()` in `act_comm.c`:**

After broadcasting the player's message to the room, iterate NPCs:

```c
for (ppl = ch->in_room->first_person; ppl != NULL; ppl = ppl->next_in_room)
{
    if (IS_NPC(ppl) && IS_SET(ppl->act, ACT_AI_DIALOGUE))
        npc_dialogue_dispatch(ppl, ch, argument);
}
```

`npc_dialogue_dispatch()` handles history lookup, expiry, serialization, and
enqueuing. It returns immediately.

**Delivery — `update_handler()` in `update.c`:**

At the top of each tick:

```c
void update_handler(void)
{
    npc_dialogue_deliver();
    /* ... existing tick logic ... */
}
```

`npc_dialogue_deliver()` locks the response queue, steals the list, unlocks,
then for each response:

1. Checks `resp->npc` and `resp->player` are non-NULL — drops the response if either is NULL
2. Validates both share a room — drops the response if not
3. Appends the assistant turn to the NPC's conversation history
4. Calls `do_say(npc, response_text)`

## Conversation History

### Storage

History is stored per-NPC, per-player on the NPC's `CHAR_DATA`:

```c
/* One turn */
typedef struct dialogue_turn {
    char role[16];       /* "user" or "assistant" */
    char content[512];
    struct dialogue_turn *next;
} DIALOGUE_TURN;

/* One player's conversation with this NPC */
typedef struct npc_conversation {
    CHAR_DATA       *player;
    time_t           last_activity;
    int              turn_count;
    DIALOGUE_TURN   *first_turn;
    DIALOGUE_TURN   *last_turn;
    struct npc_conversation *next;
} NPC_CONVERSATION;
```

A new field is added to `CHAR_DATA`:

```c
NPC_CONVERSATION *conversations;  /* NULL for players; linked list for NPCs */
```

The history is **only ever accessed by the game loop thread** — read at dispatch,
written at delivery. No additional locking is required.

### Rolling Window

The history is capped at `MAX_DIALOGUE_TURNS` (8) turns. When the cap is
reached, the oldest turn is dropped before appending the new one. This keeps the
token count sent to the model bounded and prevents unbounded memory growth.

### Expiry

At dispatch time, `last_activity` is compared to the current time. If more than
5 minutes have elapsed, the history is cleared before the new turn is added. This
gives the NPC a fresh memory when a player returns after wandering away.

### Lifecycle Example

```
[Turn 1] Player: "what do you sell?"
  History sent to model: []
  NPC replies: "Weapons and armor, traveler."
  History now: [user: "what do you sell?"] [assistant: "Weapons and armor, traveler."]

[Turn 2] Player: "how much for the sword?"
  History sent to model: the 2 turns above
  NPC replies: "That one's 50 gold. Forged it myself."
  History now: 4 turns

[Turn 3] Player walks away for 6 minutes, comes back, says "hello"
  last_activity expired → history cleared
  NPC has no memory of prior conversation
```

### Cleanup

History is freed:
- When the NPC dies (in `death.c`)
- When the player quits or disconnects: free any `NPC_CONVERSATION` nodes on
  NPCs in `char_list` whose `player` pointer matches the departing character

## Stale Response Validation

Because the worker thread operates asynchronously, by the time a response
arrives the NPC may have been killed or the player may have quit or moved. Before
delivering any response, `npc_dialogue_deliver()` validates:

1. `resp->npc` is non-NULL (NPC still exists)
2. `resp->player` is non-NULL (player still exists)
3. Both are in the same room

Responses that fail any check are silently discarded.

Pointers are nulled out at extraction time: when `extract_char(ch)` is called,
scan the pending response queue and set `resp->npc = NULL` wherever
`resp->npc == ch`, and `resp->player = NULL` wherever `resp->player == ch`.
No unique ID field on `CHAR_DATA` is needed.

---

## Thread Safety Summary

| Resource | Owner | Protection |
|---|---|---|
| `CHAR_DATA`, all game structs | Game loop thread only | n/a (single-threaded access) |
| `NPC_CONVERSATION` history | Game loop thread only | n/a |
| Request queue (`req_head`) | Both threads | `req_mutex` + `req_cond` |
| Response queue (`resp_head`) | Both threads | `resp_mutex` |
| Queue node memory | Both threads | `malloc`/`free` (not ssm.c) |
| libcurl handles | Worker thread only | `curl_global_init` at startup |
